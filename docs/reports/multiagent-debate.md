## 아키텍처 및 알고리즘

### 핵심 개념

Multiagent Debate는 ‟여러 LLM 인스턴스가 서로의 응답에 대해 **비판(critique)과 반론(refutation)**을 주고받으면서 최종 답에 수렴"하는 방법론이다. MoA가 "여러 모델의 출력을 하나가 종합"하는 단방향 흐름이라면, Multiagent Debate는 **쌍방향 비판 루프**를 핵심으로 한다.

### 알고리즘 구조

```
[라운드 1]
  Agent A → 응답 a₁
  Agent B → 응답 b₁
  Agent C → 응답 c₁

[라운드 2]
  Agent A: "b₁과 c₁을 참고하여 a₁을 수정/유지" → a₂
  Agent B: "a₁과 c₁을 참고하여 b₁을 수정/유지" → b₂
  Agent C: "a₁과 b₁을 참고하여 c₁을 수정/유지" → c₂

[라운드 3]
  Agent A: "b₂과 c₂를 참고하여 a₂를 수정/유지" → a₃
  ... (반복)

[수렴 판정]
  모든 에이전트의 답이 동일 → 종료
  또는 최대 라운드 도달 → 다수결
```

### MoA와의 비교

| 차원 | MoA | Multiagent Debate |
|---|---|---|
| **상호작용 방향** | 단방향 (레이어 → 레이어) | 양방향 (에이전트 ↔ 에이전트) |
| **비판 여부** | ❌ (종합만) | ✅ (명시적 비판/반론) |
| **수렴 메커니즘** | 없음 (Aggregator 단일 결정) | 반복 수렴 (consensus) |
| **라운드 수** | 고정 (레이어 수) | 동적 (수렴까지) 또는 고정 |
| **피드백** | 없음 | 매 라운드 상호 피드백 |

### 핵심 결과 (논문)

| 태스크 | 단일 모델 | Multiagent Debate | 개선 |
|---|---|---|---|
| GSM8K (수학) | 77.4% | 85.2% | +7.8%p |
| MMLU (지식) | 다양 | 다수 카테고리 향상 | +3-6%p |
| Factuality | 기준 | 유의미한 향상 | - |

- 특히 **추론(reasoning)**과 **사실 검증(factuality)** 태스크에서 효과 두드러짐
- 단순 지식 회상 태스크에서는 개선 폭 작음

### 변형 및 후속 연구

#### 1. Debate with Judge

```
[Agent A 주장] vs [Agent B 주장] → [Judge가 판정]
```
- 토론자와 판정자를 분리
- 법정 토론(adversarial debate) 구조

#### 2. Society of Mind

```
[다수 에이전트가 병렬 토론]
    → [투표 or 합의]
        → [메타-에이전트가 최종 결정]
```
- 대규모 에이전트 사회 시뮬레이션
- 민주적 의사결정 모델

#### 3. SmartGPT

```
[LLM → 3개 답변 생성] → [LLM이 각 답변 비판] → [LLM이 최종 선택]
```
- 같은 모델의 여러 인스턴스가 자기 비판
- 외부 모델 불필요 (단일 모델로도 가능)

## 기술적 특징

### 수렴 조건

1. **만장일치(Unanimity)**: 모든 에이전트가 동일 답 → 종료
2. **다수결(Majority)**: 과반수 동의 → 종료
3. **라운드 제한**: 최대 N 라운드 후 다수결 또는 마지막 응답
4. **변화 없음(Stagnation)**: 연속 라운드에서 답이 변하지 않으면 종료

### 프롬프트 구조 (일반적 구현)

```
다른 에이전트들의 응답:
---
Agent A: [a₁ 전문]
Agent B: [b₁ 전문]
---

위 응답들을 검토하고:
1. 동의하는 부분과 반대하는 부분을 명시하세요
2. 반대하는 부분에 대해 근거를 제시하세요
3. 최종 답변을 제시하세요
```

### 비용 분석

- 에이전트 N개, 라운드 R개: **N × R** LLM 호출
- 각 라운드에서 입력 토큰이 증가 (이전 응답 참조)
- 라운드 3-5가 일반적으로 최적 (그 이상은 한계 수익 체감)
- MoA 대비 토큰 비용이 높음 (양방향 참조로 인한 입력 증가)

## pyreez와의 비교

| 차원 | Multiagent Debate | pyreez |
|---|---|---|
| **비판 구조** | 대칭적 (모든 에이전트가 동등) | 비대칭 (Producer→Reviewers→Leader) |
| **역할 분화** | ❌ (모든 에이전트 동일 역할) | ✅ (3가지 구별된 역할) |
| **수렴 메커니즘** | 만장일치/다수결 (명시적) | Leader 판단 기반 합의 |
| **라운드 수** | 동적 (수렴까지) | 고정 (단일 숙의 사이클) |
| **이종 모델** | 가능하나 동종이 일반적 | ✅ 핵심 (7개 프로바이더, 21개 모델) |
| **모델 선택** | ❌ | ✅ CLASSIFY→PROFILE→SELECT |
| **모델 평가** | ❌ | ✅ Bradley-Terry 14차원 |
| **MCP** | ❌ | ✅ |
| **구현 수준** | 연구 방법론 (논문 + 참조 코드) | 프로덕션 서비스 (MCP 서버) |

### 핵심 차이

Multiagent Debate는 **pyreez의 숙의 기능과 가장 직접적으로 비교 가능한 연구**다. 둘 다 "여러 모델이 서로의 출력을 비판하여 더 나은 답을 만든다"는 동일한 직관을 공유한다.

그러나 pyreez의 설계는 Multiagent Debate를 넘어서는 세 가지 차별점이 있다:

1. **역할 분화**: Debate는 모든 참가자가 동등하지만, pyreez는 Producer(초안), Reviewer(비판), Leader(합의)로 역할을 명확히 분리. 이는 의사결정 이론(Delphi method)에서 검증된 구조.

2. **이종 모델 자동 선택**: Debate는 "어떤 모델을 참가시킬지"를 수동으로 결정하지만, pyreez는 태스크를 분석하여 최적 모델 조합을 자동 선택. 12 도메인 × 62 태스크 분류 체계와 Bradley-Terry 14차원 점수를 기반.

3. **프로덕션 인터페이스**: Debate는 연구 코드 수준이지만, pyreez는 MCP 프로토콜로 표준화된 인터페이스를 제공.

## 커뮤니티 반응

- **학술적 영향**: "LLM 자기 개선(self-improvement)" 연구의 핵심 참조 논문
- **실용성 논쟁**: "비용 대비 효과가 충분한가?" — 추론/사실검증에는 효과적이나, 비용이 높음
- **MoA와의 비교**: "MoA는 효율적이고 Debate는 효과적이다" — 용도에 따른 선택
- **구현 다양성**: SmartGPT, Society of Mind 등 다양한 변형이 커뮤니티에서 등장
- **한계 인식**: "라운드가 늘어도 특정 지점 이후 개선 정체", "이미 정확한 답을 가진 모델이 토론에서 오히려 잘못된 답으로 바뀌는 경우 보고"
- **적용 사례**: 코드 리뷰(여러 모델이 서로의 코드를 리뷰), 사실 검증, 수학 문제 풀이

## 요약

Multiagent Debate는 **"LLM 간 상호 비판을 통한 응답 품질 향상"**을 실증한 핵심 연구로, pyreez 숙의 기능의 학술적 선조(academic predecessor)에 해당한다. 대칭적 토론 구조, 수렴 기반 종료, 사실성/추론 영역에서의 효과를 입증했다. pyreez는 이 개념을 **역할 분화, 자동 모델 선택, MCP 인터페이스**로 확장하여 연구 방법론에서 프로덕션 서비스로 발전시킨 구현이다.
